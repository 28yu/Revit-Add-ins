## あなたへの依頼

配布サイト `https://28yu.github.io/28tools-download/` の新機能として、**構造図PDFから柱リスト・梁リストを抽出して Excel ファイルを生成するクライアントサイドWebツール**を実装してください。

このツールは Revit アドイン Tools28 の新機能「ファミリタイプ自動作成」とペアで動作します:

```
[Web Tool]                              [Revit Add-in (Tools28)]
PDFアップロード                            生成された Excel を読込
    ↓                                       ↓
PDF.js でテキスト+座標+罫線抽出              Excel をパース
    ↓                                       ↓
表セル化（罫線駆動）                         FamilyManager.NewType() で
    ↓                                       タイプ自動生成
化け検出 → 化けはOCRフォールバック
    ↓
プレビューUI（ユーザーが列ヘッダーアサイン）
    ↓
SheetJS で Excel 生成・ダウンロード ─────→  ユーザーが手動で修正・確認 ─→
```

**Web Tool 側 = あなたの担当**。Add-in 側は別途 Tools28 リポジトリで実装します。

## 実証済み事実（重要・前提）

別セッションでサンプルPDF2件を `pdfjs-dist` で抽出済み。**この実証結果が方針の根拠**です。

### サンプルPDFと抽出結果

| ファイル | ページ | 内容 | PDF.js 抽出 |
|---|---|---|---|
| PDF1: `96492415-test_____.pdf` (131KB) | 1 | CFT充填鋼コンクリート柱リスト | ❌ **全壊** |
| PDF2: `f2a1c63f-test____...pdf` (737KB) | 1 | 基礎大梁リスト(2) | ✅ **完璧** |

### 抽出例（PDF2 page1 = 成功例）

```
"スパン" / "FG1" / "6,450" / "D25" / "6+2" / "6" / "D13[2]@150"
"スパン" / "FG7" / "8,400" / ...
"のみ" / "Xd1" / "Xd2" / "1,200" / ...
```
→ **日本語見出し・符号・寸法・配筋情報すべて正しく取れる。座標(transform[4],[5])もセル単位で正確**。

### 化けの2類型

| 類型 | 原因 | 復旧手段 |
|---|---|---|
| **A: PDF.jsで読める** | 正常なToUnicode CMap | そのまま使用 |
| **B: PDF.jsで化ける** | ToUnicode CMap自体が壊れている／無い | **OCR以外に手段なし**。Canvas に描画して Tesseract.js |

**結論**: 同一PDF内でも **ページ単位で正常/化けが混在する** ため、ページ毎に判定→分岐する設計が必須。

### 化け検出ロジック（実装済み草案）

```javascript
function isGarbled(text) {
  if (!text) return false;
  let suspicious = 0, total = 0;
  for (const ch of text) {
    const cp = ch.codePointAt(0);
    if (cp < 0x20) continue;
    total++;
    if ((cp >= 0x20000 && cp <= 0x2FFFF) ||  // CJK拡張B以降
        (cp >= 0xF900  && cp <= 0xFAFF)  ||  // CJK互換漢字
        (cp >= 0x0590  && cp <= 0x08FF)  ||  // ヘブライ/アラビア
        (cp >= 0x1000  && cp <= 0x1FFF)) {   // ミャンマー等
      suspicious++;
    }
  }
  return total > 5 && (suspicious / total) > 0.3;
}
```

PDF1 と PDF2 page3 はこの判定で確実に「化け」判定される。

```
└─────────────────────────────────────────────────────┘
```

### 表セル化アプローチ（A-2 ハイブリッド型）

```
1. PdfPig相当 = PDF.js の operatorList から罫線(line/path)抽出
2. 水平線をY座標でクラスタリング → 行境界
3. 垂直線をX座標でクラスタリング → 列境界
4. グリッド交差から矩形セル算出
5. テキストアイテムを「自分が中に入るセル」に割り当て
6. 罫線が取れないPDFは座標クラスタリングへフォールバック
```

**注意**: PDF.js の罫線抽出は `page.getOperatorList()` から `OPS.constructPath` / `OPS.stroke` を拾う方式。`pdf2json` や `pdf.js-extract` の高レベルAPIでは罫線が取れないので、低レベル API 直接利用が必要。

## 次のステップ（推奨順序）

このセッションでは机上検討+実証で **A-1 (PDF種類判別) + A-2 (テキスト座標抽出基礎技術)** までを完了済み。
次は以下の順序で進めてください:

### Step 1: Excelスキーマ確定（最優先・D-5）

「②Excel→ファミリタイプ作成」工程と完全に整合する Excel フォーマットを先に決める。
これが決まらないと表セル化のゴールがブレる。

**確定すべき項目（柱・梁それぞれ）**:
- シート構成（1シート vs カテゴリ別シート）
- 列定義: 符号 / 階範囲 / 断面B / 断面D / 主筋 / 帯筋 / コンクリート強度 / 鉄骨断面名 / 備考
- 配筋情報の格納形式（生文字列 `4-D22` のまま vs パース後 `本数=4, 径=D22`）
- ヘッダー行の固定構造（Tools28 側パーサーが期待する形式）

**ユーザーと協議すべきこと**:
- 既存の社内 Excel フォーマットがあるか
- 柱・梁・S造/RC造で別シートにするか共通か

### Step 2: 罫線抽出プロトタイプ（A-2 検証）

PDF2 page1（基礎大梁リスト）を題材に、**実際に罫線を抽出して表セルが構築できるか**を最小コードで実証する。
具体的には:
- `page.getOperatorList()` から線分抽出
- 行・列境界を算出
- セルごとにテキスト割り当て
- CSV としてダンプ

これが成立すれば本実装フェーズへ。詰まれば座標クラスタリング型に方針変更。

### Step 3: OCRフォールバック検証

PDF1（柱リスト・全壊）を Canvas にレンダリング → Tesseract.js で OCR して、**実用精度（特に数字・英字 `D25` `4-D22` `500×500` 等）**が出るかを確認。

### Step 4 以降

- 表領域選択UI
- 列ヘッダーアサインUI
- 化けセル手動編集
- Excel書き出し（SheetJS）
- 配布サイト統合（既存スタイルと整合）
